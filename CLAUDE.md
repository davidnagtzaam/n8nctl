## Project Overview

**n8nctl** is a production-ready deployment tool for n8n (workflow automation platform) using Docker Compose. It provides interactive setup, backup/restore, migrations, health monitoring, and scaling capabilities with both beautiful UI (via `gum`) and fallback modes.

## Architecture

### Service Topology
```
Internet → Traefik (HTTPS) → n8n-web (UI/webhooks)
                           ↓
                      Redis Queue ← n8n-worker (scalable)
                           ↓
                      PostgreSQL (local or external)
```

- **Traefik**: Reverse proxy with automatic Let's Encrypt HTTPS
- **Redis**: Bull queue backend for distributed execution
- **n8n-web**: Main process handling UI and webhooks (N8N_DISABLE_PRODUCTION_MAIN_PROCESS=false)
- **n8n-worker**: Background job processors (N8N_DISABLE_PRODUCTION_MAIN_PROCESS=true), horizontally scalable
- **PostgreSQL**: Either local (compose.local-db.yaml) or external

## Common Development Commands

```bash
# Validate Docker Compose files
make validate

./scripts/test.sh
# or
n8nctl test

./scripts/validate-env.sh
```

### Service Management
```bash
# Start services (detects local vs external DB automatically)
make up              # External DB
make up-local        # Local PostgreSQL
n8nctl start         # Auto-detects

# View logs
make logs            # All services
n8nctl logs          # All services
n8nctl logs n8n-web  # Specific service

# Check status
make status
n8nctl status

# Restart services
make restart
n8nctl restart
```

### Development & Debugging
```bash
# Shell access to containers
make shell-web
make shell-worker
n8nctl shell n8n-web

# Execute commands in containers
n8nctl exec n8n-web n8n --version

# Scale workers
make scale-workers COUNT=5
n8nctl scale 5
```

### Operations
```bash
# Backup system
sudo ./scripts/backup.sh
n8nctl backup

# Restore from backup
sudo ./scripts/restore.sh /tmp/n8n-backup-<timestamp>.tgz
n8nctl restore /tmp/backup.tgz

# Safe migration (with auto-backup)
sudo ./scripts/migrate.sh
n8nctl migrate

# Quick upgrade
sudo ./scripts/upgrade.sh
n8nctl upgrade
```

## Important Technical Details

### UI Library (lib/lib-ui.sh)
- **Dual-mode**: Uses `gum` (Charm.sh) for enhanced UI if available, otherwise falls back to basic colors
- **Detection**: Automatically detects gum availability and TTY
- **Config**: User preferences stored in `~/.config/n8nctl/config` (XDG-compliant)
- **Functions**: All print/prompt functions check `$USE_GUM` flag and adapt behavior
- When writing new scripts that need UI, source lib-ui.sh and use its functions

### Configuration Management (lib/lib-config.sh)
- **Location**: `~/.config/n8nctl/config`
- **Functions**: `config_get`, `config_set`, `config_exists`, `config_init`
- Independent module that can be sourced separately
- Handles platform-specific sed differences (macOS vs Linux)

### Docker Compose Strategy
- **Base**: `compose.yaml` (assumes external PostgreSQL)
- **Overlay**: `compose.local-db.yaml` (adds local PostgreSQL)
- **Usage**: `docker compose -f compose.yaml -f compose.local-db.yaml up -d` for local DB
- Scripts auto-detect which mode is active by checking running containers

### Environment Configuration
- `.env.template`: Template with placeholders
- `.env`: Generated by init.sh, contains actual values
- Scripts validate for placeholders like "CHANGE_ME", "your_", "example.com"
- Contains both n8n config and PostgreSQL credentials

### Queue Mode Architecture
- **EXECUTIONS_MODE=queue**: Required for scaling
- **n8n-web**: Handles web UI, API, and webhooks (main process enabled)
- **n8n-worker**: Only processes queue jobs (main process disabled)
- Both share the same PostgreSQL database and Redis queue
- Workers share n8n-data volume for local binary storage

### Backup System
- Creates timestamped tar.gz archives containing:
  - PostgreSQL dump
  - n8n workflows and credentials
  - Configuration files (.env)
  - Metadata (timestamp, version info)
- Verification step after backup creation
- Configurable retention via BACKUP_RETENTION_LOCAL

### Script Standards
All scripts follow this pattern:
```bash
#!/usr/bin/env bash
set -euo pipefail  # Exit on error, undefined vars, pipe failures

# Load shared UI library
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SCRIPT_DIR/../lib/lib-ui.sh"
```

## Making Changes

### When Modifying Scripts
1. **Always source lib-ui.sh** for consistent UI
2. **Use `set -euo pipefail`** for safety
3. **Test both with and without gum** installed
4. **Validate with ShellCheck** if available
5. **Test in both local and external DB modes**

### When Adding New Features
1. Add script to `scripts/` directory
2. Add command to `scripts/n8nctl` CLI wrapper
3. Add Makefile target if appropriate
4. Update README.md with usage examples
5. Test the new functionality

### When Changing Docker Compose
1. Validate with `docker compose config`
2. Test both compose.yaml alone and with local-db overlay
3. Ensure healthchecks are defined for new services
4. Update compose.local-db.yaml if database-related

### When Modifying UI Library
- Test with `USE_GUM=true` and `USE_GUM=false` paths
- Ensure prompts work in non-interactive mode (for automation)
- Export new functions at end of lib-ui.sh

## Code Quality Standards

### Production-Ready Code
- **Robustness**: All code must handle edge cases, validate inputs, and fail gracefully with clear error messages
- **Security**: Follow secure coding principles - validate user input, sanitize paths, use proper permissions (600 for sensitive files)
- **Error Handling**: Use `set -euo pipefail` and trap errors appropriately; provide actionable error messages

### Naming Conventions
- **Variables**: Use descriptive lowercase with underscores (e.g., `backup_destination`, `postgres_host`)
- **Functions**: Use descriptive verb_noun format (e.g., `collect_domain_config`, `validate_backup_integrity`)
- **Constants**: Use uppercase with underscores (e.g., `SCRIPT_DIR`, `N8N_HOST`)
- **Consistency**: Follow existing patterns in the codebase

### Comments
- **Purpose-driven**: Explain *why* code exists, not *what* it does
- **Context**: Document non-obvious decisions, workarounds, or business logic
- **Avoid**: Changelog-style comments ("changed this here"), obvious statements ("now doing X"), or self-referential phrases like "professional code"

Example of good comments:
```bash
# Generate encryption key using OpenSSL to ensure cryptographic strength
# Note: tr removes characters that could break shell interpolation
encryption_key=$(openssl rand -base64 32 | tr -d "=+/")

# Redis persistence disabled for queue use case - data is transient
# and losing it on restart only affects in-flight jobs
command: ["redis-server", "--save", "", "--appendonly", "no"]
```

### Modularity & Abstraction
- **Reusable Libraries**: Keep shared functionality in `lib/` (e.g., UI functions in lib-ui.sh, config management in lib-config.sh)
- **Single Responsibility**: Each script and function should do one thing well
- **Loose Coupling**: Changes to one module (e.g., UI library) should not require changes to business logic in scripts
- **Clear Interfaces**: Library functions should have clear inputs/outputs and work independently

When creating or modifying code:
- If adding UI interactions, use lib-ui.sh functions rather than raw echo/read
- If storing configuration, use lib-config.sh rather than directly writing files
- If adding new shared functionality, consider if it belongs in a library module

### Scope Discipline
**IMPORTANT**: Never add or modify code outside the scope of what was explicitly requested.

Before making changes beyond the original request:
1. **Stop and ask**: Share your idea with the user
2. **Explain the suggestion**: What enhancement you're proposing and why
3. **Wait for approval**: Do not proceed without explicit permission

This applies to:
- Adding new functions or features
- Creating new documentation files
- Introducing new folder structures
- Adding new dependencies or frameworks
- Refactoring code that wasn't mentioned in the request

If you notice opportunities for improvement, mention them but don't implement them unless asked.

### Tone & Communication
- **User-facing messages**: Be clear, helpful, and a bit playful with personality (inspired by gum's aesthetic and Claude Code's tone)
- **Error messages**: Friendly but informative - help users understand what went wrong and how to fix it
- **Success messages**: A little celebration is fine (✨, ✅) but stay grounded
- **Documentation**: Professional and precise, with a human touch

Examples of good tone:
```bash
print_success "Database backup complete! Your data is safe and sound."
print_warning "Hmm, Redis seems to be taking a nap. Let's give it a moment..."
print_error "Oops! Couldn't connect to PostgreSQL. Check your DATABASE_URL and try again."
print_info "Scaling workers to 5 replicas - time to parallel process! 🚀"
```

Avoid:
- Overly casual language in error messages
- Corporate jargon or buzzwords
- Excessive emoji usage
- Condescending tone

## Storage Options

- **Local**: `N8N_DEFAULT_BINARY_DATA_MODE=default` (uses n8n-data volume)
- **S3-compatible**: `N8N_DEFAULT_BINARY_DATA_MODE=s3` (requires S3_* env vars)
- Both modes supported, configured during init.sh setup

## Security Considerations

- `.env` files set to 600 permissions (owner read/write only)
- Encryption keys generated with `openssl rand`
- No new privileges for Docker containers (`no-new-privileges:true`)
- Traefik provides security headers and rate limiting
- Automatic HTTPS via Let's Encrypt
